INTRENSIC-VALUATOR — IMPLEMENTATION PLAN
================================================================================

1. EXECUTIVE SUMMARY
================================================================================

The Intrensic Valuator is an agentic AI system that values publicly traded stocks
and outputs a Buy/Sell/Hold recommendation. It uses a top-down fundamental
analysis approach: Country (macro) → Industry (sector) → Company (firm-specific).
Each layer is analyzed by a specialized AI agent, and the qualitative + quantitative
outputs feed into a Discounted Cash Flow (DCF) model to compute intrinsic value.

The final recommendation compares intrinsic value to market price with a margin of
safety threshold, and includes a confidence score based on forecast uncertainty.

--------------------------------------------------------------------------------
2. THE TOP-DOWN VALUATION FRAMEWORK
================================================================================

2.1 Why Top-Down?
--------------------------------------------------------------------------------
Most automated valuation tools start and end with company financials — they
ignore where the company operates and what industry it's in. But a tech company
in India faces fundamentally different risks and opportunities than a tech company
in the US. The top-down approach captures these structural differences:

  COUNTRY  →  Defines the baseline cost of capital (sovereign risk, inflation,
              currency stability, GDP trajectory)

  INDUSTRY →  Defines the growth ceiling and competitive risk (TAM, Porter's
              5 Forces, regulation, technological disruption risk)

  COMPANY  →  Defines firm-specific execution ability (management quality,
              competitive moat, financial health, growth track record)

Each layer ADJUSTS the numbers from the layer above it. This cascading adjustment
is the theoretically correct way to value a company in context.

2.2 The Integration Point: WACC
--------------------------------------------------------------------------------
All three layers converge on the Weighted Average Cost of Capital (WACC):

  WACC = f(Country_Risk, Industry_Beta, Company_Specific_Risk)

Country risk sets the risk-free rate floor (via sovereign bond yields / CDS
spreads). Industry risk determines beta (systematic risk relative to market).
Company-specific risk adds a final premium for things like small size, high
leverage, or operational complexity.

This means every risk identified at any layer automatically makes the stock
cheaper (higher discount rate = lower present value of future cash flows).
This is the fundamental mechanism linking qualitative analysis to quantitative
valuation.

All API keys are stored in a `.env` file at the project root. This is loaded
by python-dotenv at application startup. The `.env.example` file serves as a
template showing all required and optional keys.

4. DATA SOURCES — WHAT, WHY, AND HOW
================================================================================

4.1 Primary Data Sources (FREE — no API key needed for basics)
--------------------------------------------------------------------------------

A. yfinance (Yahoo Finance)
   - What: Stock price, income statement, balance sheet, cash flow statement,
          shares outstanding, beta, industry/sector classification, peers
   - Why: Most comprehensive free source for US stocks. Covers 99% of what
          we need for financial statement data.
   - How: Pull FCF from cash flow statement line items
          (Operating Cash Flow - Capital Expenditure), NOT from info dict.
          The info['freeCashflow'] field is known to be unreliable in yfinance.
   - Rate limits: None officially, but use caching to be respectful.

B. SEC EDGAR (XBRL API)
   - What: Raw 10-K and 10-Q filings with structured financial data
   - Why: Most accurate and auditable source. Used as verification layer
          and as the primary source for true Diluted Shares Outstanding.
   - How: HTTP GET to SEC EDGAR API, parse XBRL-JSON or use edgartools library.
          Maps XBRL concepts (us-gaap taxonomy) to our financial model fields.
   - Rate limits: 10 requests/second. Very generous.

C. FRED (Federal Reserve Economic Data via fredapi)
   - What: 10-Year US Treasury yield (risk-free rate), GDP growth, CPI/inflation,
          unemployment rate, federal funds rate
   - Why: The gold standard for US macroeconomic data. Used to calibrate the
          risk-free rate and macro growth assumptions.
   - How: fredapi Python library. Requires a free FRED API key (sign up at
          research.stlouisfed.org).
   - Rate limits: 120 requests/minute with API key.

D. World Bank API (via wbgapi)
   - What: GDP growth, inflation, population, governance indicators by country
   - Why: Needed for non-US stocks to calibrate country-level macro assumptions.
          Governance indicators serve as proxies for political stability.
   - How: wbgapi Python library. No API key required.
   - Rate limits: Generous, no strict limit documented.

4.2 Secondary Data Sources (Freemium — used as fallback/enrichment)
--------------------------------------------------------------------------------

E. Financial Modeling Prep (FMP)
   - What: Analyst consensus estimates, pre-computed DCF, treasury rates,
          market risk premium data, enterprise values, peer trading multiples.
   - Why: Analyst estimates provide a useful sanity check against our AI-generated
          projections. Essential fallback when yfinance peer data returns NaNs.
   - How: REST API with API key. Free tier: 250 requests/day.
   - URL: https://financialmodelingprep.com/developer/docs/

F. Alpha Vantage
   - What: Income statement, balance sheet, cash flow, forex rates
   - Why: Alternative data source if yfinance has gaps (especially for
          international stocks).
   - How: REST API with API key. Free tier: 25 requests/day.
   - URL: https://www.alphavantage.co/

4.3 Data Caching Strategy
--------------------------------------------------------------------------------

A dual-layer caching system prevents redundant API calls during development and
repeated valuation runs. Both layers are built and verified.

Financial data changes on known schedules:
  - Stock prices: Daily (or real-time, but daily is enough for valuation)
  - Financial statements: Quarterly (10-Q) and Annually (10-K)
  - Macro data: Monthly (CPI, employment) or Quarterly (GDP)

Caching TTLs (enforced at application layer):
  TTL_PRICE      = 86_400s  (1 day)   — stock quotes
  TTL_FINANCIALS = 604_800s (7 days)  — income stmt, balance sheet, cash flow
  TTL_MACRO_RATES= 604_800s (7 days)  — treasury yields, interest rates
  TTL_MACRO_GDP  = 2_592_000s (30 days) — GDP growth, CPI, macro aggregates
  TTL_ESTIMATES  = 604_800s (7 days)  — analyst consensus estimates
  TTL_DEFAULT    = 86_400s  (1 day)   — fallback

Cache key convention: "{source}:{identifier}:{data_type}"
  Examples: "yfinance:AAPL:cash_flow", "fred:GDP:quarterly",
            "worldbank:IN:gdp_growth", "fmp:AAPL:analyst_estimates"

Layer 1 — Application Cache (src/data/data_cache.py)
  Class: DataCache(db_path) — SQLite-backed, thread-safe, TTL-driven.

  Schema:
    CREATE TABLE cache_entries (
        key        TEXT PRIMARY KEY,
        value      TEXT NOT NULL,     -- JSON-serialized
        expires_at REAL NOT NULL,     -- Unix timestamp
        created_at REAL NOT NULL
    );

  Design decisions:
    - JSON serialization with custom encoder for numpy scalars/arrays,
      pandas DataFrames (orient="split"), and pandas Series
    - Lazy eviction: expired entries detected and deleted on get() —
      no background thread required
    - WAL journal mode for concurrent reads
    - threading.Lock around writes; reads are lock-free
    - Hit/miss/expiration counters exposed via stats()

  API: get(key) → Any|None, set(key, value, ttl_seconds), delete(key),
       clear(), expire() → count, stats() → dict

Layer 2 — HTTP Cache (src/data/http_cache.py)
  Thin wrapper around requests_cache.CachedSession.

  URL-pattern TTLs:
    *.yahoo.com/*              → 1 day
    data.sec.gov/*, www.sec.gov/* → 1 day
    api.stlouisfed.org/*       → 7 days
    api.worldbank.org/*        → 30 days
    financialmodelingprep.com/* → 7 days
    www.alphavantage.co/*      → 7 days

  Settings:
    - Backend: SQLite (separate DB from application cache)
    - stale_if_error=True (serve stale cache if the remote is down)
    - allowable_codes=(200, 301, 302, 404) (cache 404s briefly)
    - Singleton via get_http_session()

  Fetcher modules call session.get() directly — caching is transparent;
  they don't need to interact with the application cache unless they
  want to cache post-processed results.

Verified behavior (all tests passed):
  - set/get with dict, list, scalar values
  - TTL expiry (lazy eviction on read)
  - None on miss and on expired key
  - NumPy float64, int64, bool_, ndarray round-trip
  - Pandas DataFrame and Series round-trip
  - Stats counters (hits, misses, expirations, hit rate, DB size)
  - Explicit expire(), delete(), clear()

5. THE DCF VALUATION ENGINE
================================================================================

5.1 Free Cash Flow Calculation
--------------------------------------------------------------------------------
Starting point — pull from the most recent fiscal year or trailing twelve months:

  FCF = Operating_Cash_Flow − Capital_Expenditure

  Where:
    Operating Cash Flow = Line item from Cash Flow Statement
    Capital Expenditure = Line item from Cash Flow Statement
                          (often labeled "Purchase of Property, Plant & Equipment"
                          or "Capital Expenditure")

  If OCF > Net Income consistently → Quality earnings (positive signal)
  If Capex < Depreciation consistently → Underinvesting (negative signal)

5.2 Revenue & FCF Projection (Years 1-5 explicit, then terminal)
--------------------------------------------------------------------------------
The projection embeds the top-down analysis directly into the numbers:

  Revenue_Year_N = Revenue_Base × Π(i=1 to N) (1 + Growth_Rate_i)

  Where Growth_Rate_i starts at the company-specific rate (from Company Agent),
  then systematically decays sequentially toward the industry rate (from Industry Agent),
  and finally toward the terminal growth rate (from Country Agent).

  To avoid mathematical discontinuity jumps between phases while preserving the
  three-layer top-down framework, a continuous piecewise linear decay (3-knot spline)
  is applied across the 5-year explicit horizon:

    Knot 0 (Year 0): Company_Growth        ← Firm-specific starting point
    Knot 1 (Year 3): Industry_Growth       ← Industry-anchored midpoint
    Knot 2 (Year 5): Terminal_Growth       ← Long-run macro steady state

  The growth rate for each projection year i is:

    For i = 1, 2, 3:
      Growth_Rate_i = Company_Growth + (i / 3) × (Industry_Growth − Company_Growth)

    For i = 4, 5:
      Growth_Rate_i = Industry_Growth + ((i − 3) / 2) × (Terminal_Growth − Industry_Growth)

  At i = 3, both segments evaluate to Industry_Growth — the transition is seamless.
  This embeds the top-down logic directly: the company's growth converges to the
  industry norm by year 3, then the industry rate decays toward the terminal
  long-run GDP growth rate by year 5. No one-step jump from Company to Terminal
  ever occurs; Industry acts as the structurally necessary intermediate anchor.

  FCF_Year_N = Revenue_Year_N × FCF_Margin_Year_N

  Where FCF_Margin_Year_N = Base_FCF_Margin + adjustment from company/industry analysis.

5.3 WACC Calculation (THE INTEGRATION POINT)
--------------------------------------------------------------------------------
This is where ALL three layers of analysis converge. Every risk identified in
the top-down analysis feeds into WACC, which determines how aggressively we
discount future cash flows.

  WACC = (E/V) × Cost_of_Equity + (D/V) × Cost_of_Debt × (1 − Tax_Rate)

  Where:

  A) Cost of Equity (CAPM with Country Risk Premium):
     Cost_of_Equity = Risk_Free_Rate + Industry_Beta × Equity_Risk_Premium
                      + Country_Risk_Premium + Size_Premium

     Risk_Free_Rate:
       - Source: 10-Year US Treasury yield from FRED
       - For non-US stocks: Local 10-year government bond yield
       - Market convention: Use the yield on the longest-dated government bond
         of the country where the company operates

     Equity Risk Premium (ERP):
       - Default: 5.0% for developed markets, higher for emerging
       - Source: Can be estimated from FMP or use Damodaran's published ERP
       - Typical ranges: US 4.5-5.5%, Emerging Markets 6-9%

     Industry Beta:
       - Source: Calculate from peer group stock returns vs S&P 500, or
         use Damodaran's industry beta tables (published by sector)
       - If using peer betas: Median unlevered beta of 5-10 industry peers,
         then relevered to the company's own D/E ratio
       - Hamada formula: β_levered = β_unlevered × [1 + (1 − Statutory_Tax_Rate) × (D/E)]
         *NOTE: Statutory tax rate is utilized here instead of Effective tax rate to avoid
         artificially over-levering or under-levering companies due to short-term tax anomalies.

     Country Risk Premium (CRP):
       - Source: Country Agent analysis
       - Measurement approaches:
         a) Sovereign CDS spread vs US CDS (market-based, preferred)
         b) Sovereign bond yield spread vs US Treasury (for countries with bonds)
         c) Damodaran's country risk premiums (published annually, covers 190+ countries)
         d) Governance indicators mapped to risk premiums
       - Typical values: Germany 0%, US 0%, Brazil 2-4%, India 2-3%,
         China 1-2%, Argentina 8-15%, Turkey 5-10%
       - This is the SINGLE LARGEST source of cross-country valuation differences

     Size Premium:
       - Additional return required for small-cap stocks
       - Source: CRSP Decile premiums or Duff & Phelps size premium study
       - Typical: 0% for mega-cap, 1-3% for small-cap, 3-6% for micro-cap
       - Only apply when market cap < $2B

  B) Cost of Debt:
     Cost_of_Debt = (Risk_Free_Rate + Credit_Spread) × (1 − Statutory_Tax_Rate)

     Credit_Spread:
       - Estimated from company's credit rating, interest coverage ratio,
         or debt/EBITDA
       - Moody's/S&P rating → spread mapping tables are well-documented
       - If no public rating: Estimate synthetic rating from financial ratios
         - Interest Coverage Ratio > 8.5 → AAA (~0.7% spread)
         - ICR 6.5-8.5 → AA (~1.0%)
         - ICR 5.5-6.5 → A (~1.3%)
         - ICR 4.25-5.5 → BBB (~2.0%)
         - ICR 3.0-4.25 → BB (~3.5%)
         - ICR 2.5-3.0 → B (~5.0%)
         - ICR < 2.5 → CCC+ (~8%+)

     Tax Rate:
       - Use Statutory Corporate Tax Rate for the base country to ensure stability in the
         long-term structural cost of capital components.

  C) Capital Structure Weights (E/V and D/V):
     - E/V = Market_Cap / (Market_Cap + Total_Debt_Proxy)
     - D/V = Total_Debt_Proxy / (Market_Cap + Total_Debt_Proxy)
     - Market Cap = Current Price × Diluted Shares Outstanding
     - Total Debt Proxy = Short-term Debt + Long-term Debt (from balance sheet)
       *NOTE: Book value of debt is used as a pragmatic and industry-standard proxy 
       for the market value of debt since true debt market valuations are untraded/unobtainable.

5.4 Terminal Value
--------------------------------------------------------------------------------
The terminal value typically represents 60-80% of total enterprise value in a
DCF model. Small changes in assumptions produce wildly different results.
We calculate TWO methods and use the average:

  Method 1: Perpetuity Growth (Gordon Growth Model)
    TV = FCF_Year_5 × (1 + g) / (WACC − g)
    Where g = terminal growth rate ≈ long-term nominal GDP growth of the country
    (capped at risk-free rate). Typically 2.0-3.5% for developed markets.

  Method 2: Exit Multiple
    TV = EBITDA_Year_5 × Industry_Median_EV/EBITDA_Multiple
    This serves as a market-based sanity check on the perpetuity method.

  Final TV = Average of both methods (or weighted: 70% perpetuity, 30% exit)

5.5 Present Value Calculation
--------------------------------------------------------------------------------
  Enterprise_Value =  Σ(t=1 to 5) FCF_t / (1 + WACC)^t
                      + Terminal_Value / (1 + WACC)^5

  Equity_Value = Enterprise_Value
                 − Total_Debt
                 + Cash_and_Cash_Equivalents
                 − Minority_Interest
                 + Investments_in_Associates

  Intrinsic_Value_Per_Share = Equity_Value / Diluted_Shares_Outstanding
  *NOTE: Diluted Shares Outstanding must be validated via SEC layer to avoid basic-share bias.

5.6 Relative Valuation (Sanity Check)
--------------------------------------------------------------------------------
DCF is sensitive to assumptions. We cross-validate with relative valuation:

  P/E Approach:   Fair_Price = Median(Peer_Group_P/E) × Company_TTM_EPS
  EV/EBITDA:      Fair_EV = Median(Peer_EV/EBITDA) × Company_EBITDA
  P/B Approach:   Fair_Price = Median(Peer_P/B) × Company_Book_Value_Per_Share

  Peer group selection:
    - Same GICS industry/sub-industry
    - Similar market cap (±50%)
    - Similar revenue size
    - Min 5 peers, ideally 8-12
    - Fallback: If yfinance fields return NaNs, automatically call FMP/Alpha Vantage API

  Final check: If DCF intrinsic value deviates >50% from relative valuation
  median, flag a "model disagreement" warning and widen confidence interval.

5.7 Scenario Analysis & Monte Carlo
--------------------------------------------------------------------------------
  Scenario 1: Bull Case
    - Revenue growth: +20% above base
    - FCF margin: +200 bps
    - WACC: −100 bps
    - Terminal growth: +50 bps

  Scenario 2: Base Case
    - All inputs at expected values from agent analysis

  Scenario 3: Bear Case
    - Revenue growth: −20% below base
    - FCF margin: −200 bps
    - WACC: +150 bps
    - Terminal growth: −50 bps

  Monte Carlo Simulation (10,000 iterations):
    - Vary: Revenue growth (normal dist), FCF margin (normal dist),
      WACC (normal dist), terminal growth (normal dist)
    - Output: Distribution of intrinsic values → mean, median, std dev,
      10th percentile, 90th percentile
    - The std dev feeds into confidence scoring

  Final Fair Value Range: [10th percentile, 90th percentile]

5.8 Assumption Validation Layer (Pre-DCF Guardrail)
--------------------------------------------------------------------------------
Before assumptions from the AI agents enter the DCF engine, they pass through
a deterministic validation layer. This prevents unrealistic or unsupported
inputs from producing misleading valuations.

  For each assumption (revenue growth, FCF margin, WACC components, terminal
  growth rate), the validation layer checks:

  A) Historical Range Check:
     Compare the generated assumption against the company's own 5-year
     historical range. If the generated value falls outside [Min_Historical,
     Max_Historical], flag a warning with magnitude of deviation.

  B) Industry Benchmark Check:
     Compare against industry median and interquartile range. A growth
     assumption 2× the industry median requires strong justification.

  C) Statistical Confidence:
     Assign a confidence band to each assumption:
       - GREEN:  Within 1σ of historical mean — no flag
       - AMBER:  1-2σ from historical mean — documented but accepted
       - RED:    >2σ from historical mean — requires explicit override flag
                 and is capped at 2σ unless the agent provides specific,
                 verifiable evidence (new product launch, market entry)

  D) Override Flag:
     If an assumption exceeds bounds, it is NOT silently rejected. Instead:
       1. The assumption is capped at the boundary value
       2. An override flag is raised with the original value and the cap
       3. The flag appears in the PDF report as a transparency note
       4. Confidence score is penalized proportionally

  Example:
    Company 5-year revenue growth range: 3% – 9%
    Agent-generated growth assumption: 17%
    → RED flag: capped at 9% (2σ upper bound)
    → Report note: "Revenue growth assumption capped at 9%. Agent initially
       suggested 17% based on [cited evidence]. Growth rate capped to 2σ
       historical ceiling. Valuation uses 9%."

  This layer ensures the DCF engine never sees unvalidated inputs, regardless
  of what the LLM agents produce.

6. AGENTIC AI WORKFLOW (LangGraph Implementation)
================================================================================

6.1 Why Agentic AI?
--------------------------------------------------------------------------------
Traditional automated valuation uses hard-coded rules. But financial analysis
requires:
  - Interpreting nuanced qualitative information (earnings call tone,
    management track record, competitive dynamics)
  - Adapting to unusual situations (turnarounds, M&A, accounting changes)
  - Synthesizing information across multiple dimensions and sources
  - Applying judgment to edge cases where formulas break

LLM agents can do all of this — IF you give them the right tools, prompts, and
structured output formats. They don't replace the quantitative model; they feed
its inputs.

6.2 Orchestrator Architecture (LangGraph with Sequential Fallback)
--------------------------------------------------------------------------------
We use LangGraph to build a directed graph of agent nodes. Each node is an
LLM call with specific tools and a structured output schema.

  State flows:
    START → CountryAgent → IndustryAgent → CompanyAgent
          → ValuationEngine → RecommendationAgent → END

  CRITICAL ROBUSTNESS GUARDRAIL:
  If a graph compilation or run execution runtime failure occurs (such as versioning
  incompatibilities, missing state keys, or LangGraph infrastructure anomalies),
  the system executes a **Sequential Fallback Mechanism**. This runs each node 
  function as a standard Python execution chain, passing the unified state 
  dictionary sequentially down the pipeline, guaranteeing absolute fault tolerance.

  Each node:
    1. Receives the full state (all prior analysis)
    2. Has access to data-fetching tools (yfinance, FRED, web search)
    3. Produces structured output (Pydantic model) that merges into state
    4. Passes enriched state to next node

  The ValuationEngine node is NOT an LLM call — it's pure Python computation
  that takes the structured outputs from the three analysis agents and runs
  the DCF model. This ensures the math is always correct and reproducible.

6.3 Agent-Specific Designs
--------------------------------------------------------------------------------

  AGENT 1: Country/Macro Agent
  Inputs: Company name, ticker, identified country
  Tools: FRED API, World Bank API, web search for current events
  Analysis:
    - GDP growth trend (past 5 years + forward outlook)
    - Inflation & interest rate environment
    - Currency stability and exchange rate trends
    - Political stability (World Bank governance indicators)
    - Central bank policy trajectory
    - Sovereign credit rating and CDS spreads
    - Key country-specific risks (sanctions, capital controls, etc.)
  Outputs:
    - country: str
    - gdp_growth_forecast: float (nominal, next 5 years)
    - inflation_forecast: float
    - political_stability_score: float (0-10)
    - country_risk_premium: float (basis points, 0-1500)
    - currency_risk_adj: float (additional discount, 0-500 bps)
    - macro_narrative: str (1-2 paragraph summary)

  AGENT 2: Industry Agent
  Inputs: Company industry/sector, country context
  Tools: yfinance (peer data), web search, SEC EDGAR (industry filings)
  Analysis:
    - Total Addressable Market (TAM) and growth rate
    - Industry lifecycle stage (emerging, growth, mature, declining)
    - Porter's 5 Forces: rivalry, barriers to entry, supplier power,
      buyer power, substitutes
    - Regulatory environment and upcoming changes
    - Technological disruption risk
    - Cyclicality and sensitivity to macro factors
    - Peer group identification (5-10 closest competitors)
    - Industry average financial metrics (margins, growth, leverage)
  Outputs:
    - industry: str
    - industry_growth_rate: float (revenue CAGR next 5 years)
    - industry_beta: float (unlevered)
    - competitive_intensity_score: float (0-10, 10 = cutthroat)
    - regulatory_risk_score: float (0-10)
    - disruption_risk_score: float (0-10)
    - peer_tickers: list[str]
    - industry_fcf_margin_avg: float
    - industry_narrative: str

  AGENT 3: Company Agent
  Inputs: Ticker, financial statements, industry context
  Tools: yfinance, web search, SEC EDGAR
  Analysis:
    - Revenue growth trend (5-year historical + analyst estimates)
    - Profitability: Gross margin, operating margin, net margin trends
    - Free cash flow generation and quality
    - Balance sheet health: Debt/Equity, Current Ratio, Interest Coverage
    - Return on Invested Capital (ROIC) vs WACC (value creation test)
    - Management quality assessment (from earnings calls, track record,
      capital allocation history)
    - Competitive moat assessment: Brand, network effects, switching costs,
      cost advantage, intangible assets
    - Growth drivers: New products, geographic expansion, pricing power,
      acquisitions
    - Risk factors: Idiosyncratic risk factors only (customer concentration, 
      key person risk, litigation, product obsolescence).
      *CRITICAL WORKFLOW GUARDRAIL: The Company Agent prompt explicitly forbids
      penalizing or adding premium for systemic macro/country-level problems 
      (e.g. general inflation or country risks) to eliminate Double-Counting Risk.
    - Financial statement red flags: Aggressive revenue recognition,
      frequent one-time charges, growing DSO, inventory build
  Outputs:
    - revenue_growth_forecast: float (next 5-year CAGR)
    - fcf_margin_forecast: float (steady-state margin)
    - moat_width_score: float (0-10)
    - management_quality_score: float (0-10)
    - financial_health_score: float (0-10)
    - company_specific_risk_premium: float (basis points, 0-500)
    - roic: float
    - debt_to_equity: float
    - company_narrative: str

  AGENT 4: Valuation Engine (Pure Python — no LLM)
  Inputs: All structured outputs from above + yfinance financial data
  Computation:
    1. Validate all agent assumptions through Assumption Validation Layer
       (Section 5.8) — historical range, industry benchmark, statistical
       confidence checks with GREEN/AMBER/RED flagging
    2. Pull FCF from cash flow statement
    3. Project FCF for 5 years using validated growth rates from agents
    4. Calculate WACC using CRP, beta, size premium
    5. Calculate terminal value
    6. Discount to present value
    7. Calculate relative valuation comparables
    8. Run Monte Carlo simulation (if enabled)
    9. Blend DCF and relative valuation (70/30)
  Outputs:
    - intrinsic_value_per_share: float
    - fair_value_low: float (10th percentile)
    - fair_value_high: float (90th percentile)
    - wacc: float
    - terminal_value_pct: float (% of total EV from terminal value)
    - relative_value_per_share: float
    - model_disagreement_flag: bool

  AGENT 5: Recommendation Agent
  Inputs: Valuation outputs, current price, all narratives
  Tools: None (synthesis only)
  Analysis:
    - Calculate margin of safety
    - Assess model confidence based on:
      a. Width of Monte Carlo distribution
      b. Agreement between DCF and relative valuation
      c. Data availability and quality
      d. Stability of historical financials
      e. Earnings estimate dispersion among analysts
      f. Macroeconomic uncertainty indicators
    - Assess qualitative risk factors that DO NOT override the valuation
      but DO affect the risk narrative and position sizing guidance:
      pending catalysts (FDA approval, contract wins), binary event risks,
      extreme technical factors
    - CRITICAL GUARDRAIL: The intrinsic valuation is NEVER overridden by
      qualitative factors. Qualitative observations affect risk communication,
      confidence scoring, and position sizing guidance only. The valuation
      stands on its own — the narrative explains the risks around it.
    - Generate clear, actionable recommendation with risk-aware narrative
  Outputs:
    - recommendation: Literal["BUY", "SELL", "HOLD"]
    - margin_of_safety: float (percentage)
    - confidence_score: float (0-100)
    - key_drivers: list[str] (3-5 bullet points)
    - key_risks: list[str] (3-5 bullet points)
    - position_sizing_note: str (risk-aware guidance, if applicable)
    - executive_summary: str

6.4 How Qualitative Becomes Quantitative (The Core Innovation)
--------------------------------------------------------------------------------
This is the hardest problem in automated valuation: converting text analysis
into numbers that feed the DCF model. To mitigate LLM "hallucinated precision" 
and arbitrary number assignment, agents select from discrete categorical intervals 
mapped back to fixed quantitative variables.

  CONVERSION TABLE:
  ═══════════════════════════════════════════════════════════════════════
  Qualitative Classification      →  Quantitative Parameter     Interval Mapping
  ═══════════════════════════════════════════════════════════════════════
  Country political stability     →  Country Risk Premium       [0, 250, 500, 750, 1000, 1500] bps
  Currency risk assessment        →  Additional discount        [0, 100, 250, 500] bps
  Industry competitive intensity  →  FCF margin pressure        [0, -100, -250, -500] bps
  Regulatory risk                 →  Beta adjustment            [-0.2, -0.1, 0.0, +0.1, +0.2]
  Disruption risk                 →  Terminal growth modifier   [-1.0%, -0.5%, 0.0%, +0.5%]
  Moat width                      →  Competitive adv. period    [3, 5, 7, 10] years
  Management quality              →  Execution growth factor    [-2.0%, -1.0%, 0.0%, +1.0%, +2.0%]
  Financial health                →  Credit spread Mapping      Fixed Synthetic Mapping (70-800 bps)
  ROIC vs WACC gap                →  Value creation confidence  Bounds Terminal Perpetuity
  ═══════════════════════════════════════════════════════════════════════

  Each conversion uses CALIBRATED BUCKETS — agents choose the structural category
  and justify their placement within the narrative output. The intervals are
  derived from corporate valuation literature (Damodaran, McKinsey, CFA curriculum).

  EVIDENCE-CHAIN REQUIREMENT: Every numerical adjustment must cite specific
  evidence. The agent cannot simply output "Management Quality: High → +2%
  Growth" — it must document the evidence chain:

    Example:
      Management Score: High
      Evidence:
      • Five-year ROIC averaging 28% vs industry 14% — consistent value creation
      • Conservative leverage (D/E 0.3) with disciplined buyback timing
      • Three successful $1B+ acquisitions integrated in past 5 years
      • CEO 12-year tenure with 18% annualized TSR during tenure
      → Growth Adjustment: +2.0% (top bucket, justified by sustained execution)
      → Confidence: High (multi-cycle track record, not single-year outperformance)

  This makes every assumption auditable and rebuildable — a human analyst
  should be able to read the evidence and either agree or disagree with the
  bucket selection. The adjustment is traceable through the entire pipeline.

7. RECOMMENDATION LOGIC
================================================================================

7.1 Core Decision Rule
--------------------------------------------------------------------------------
  If Intrinsic_Value > Current_Price × 1.20:
      → BUY (20%+ upside — sufficient margin of safety)
  If Intrinsic_Value < Current_Price × 0.90:
      → SELL (10%+ downside)
  Else:
      → HOLD

  The asymmetric threshold (20% upside needed vs 10% downside) reflects:
    - DCF models have an upward bias (terminal value is optimistic)
    - Transaction costs and taxes favor holding
    - Warren Buffett's principle: require a margin of safety on buys

  NO-OVERRIDE PRINCIPLE: The intrinsic valuation is never overridden by
  qualitative judgment. Binary event risks (FDA approval, contract outcomes,
  regulatory decisions) do not change the fair value estimate — they change
  the confidence level and the risk narrative communicated alongside it.
  If a stock has a pending binary catalyst, the recommendation stands but
  carries a position sizing warning and reduced confidence. This ensures
  the valuation remains objective and auditable, while the risk communication
  remains honest.

7.2 Confidence Scoring
--------------------------------------------------------------------------------
  Confidence_Score (0-100) = Weighted average of:

  A) Forecast Precision (25% weight):
     Score = max(0, 100 − (Monte_Carlo_StdDev / Intrinsic_Value × 100))
     Narrower distribution → higher confidence

  B) Model Agreement (20% weight):
     Deviation = |DCF_Value − Relative_Value| / DCF_Value
     Score = max(0, 100 − Deviation × 200)
     Both models agreeing → higher confidence

  C) Data Quality (15% weight):
     Checks: Are financials available for 5+ years? Is FCF positive?
     Are peer comparisons available (5+ peers)?
     Score: 0-100 based on data completeness

  D) Historical Stability (10% weight):
     How consistent are historical margins and growth rates?
     Score: 100 − (Coefficient_of_Variation × 100), bounded 0-100

  E) Analyst Consensus Agreement (10% weight):
     How closely do the AI-generated growth/margin assumptions align with
     analyst consensus estimates (from FMP)?
     Score: 100 − (|AI_Estimate − Consensus_Estimate| / Consensus_Estimate × 100)
     High dispersion among analysts also penalizes this score.

  F) Macroeconomic Uncertainty (10% weight):
     Measures current macro environment predictability:
       - VIX level (equity market uncertainty)
       - Fed funds rate trajectory uncertainty (rate change frequency)
       - GDP forecast dispersion among economists
     Low uncertainty → higher confidence.

  G) Assumption Validation Score (10% weight):
     How many agent assumptions passed the Validation Layer without flags?
       - All GREEN:  100
       - Some AMBER: 70
       - Any RED:    40
       - Multiple RED: 20
     This directly links confidence to input quality.

  Confidence Interpretation:
    >80: High confidence — recommendation is robust
    50-80: Moderate confidence — use position sizing
    30-50: Low confidence — wide range of outcomes; reduced position size
    <30: Very low confidence — speculative, not actionable without
         independent verification

8.3 PDF Generation Technical Approach
--------------------------------------------------------------------------------

  Library: reportlab (pure Python PDF generation)
  Why reportlab:
    - No external dependencies (no LaTeX, no wkhtmltopdf)
    - Full control over layout and typography
    - Tables with styling (colspan, rowspan, alternating colors)
    - Vector graphics support
    - Embed PNG images (for matplotlib charts)
    - Open source, BSD license

  Charts: matplotlib
    - Monte Carlo histogram → PNG → embedded in PDF
    - DCF waterfall chart (optional)
    - Scenario comparison bar chart

  Architecture:
    class PDFReportGenerator:
        def __init__(self, valuation_state: ValuationState):
            self.state = valuation_state
            self.story = []  # reportlab Platypus story (flowable list)

        def build_cover_page(self) -> list[Flowable]:
            # Logo (if available), company name, date, recommendation badge

        def build_executive_summary(self) -> list[Flowable]:
            # Company description, investment thesis, recommendation box

        def build_key_findings(self) -> list[Flowable]:
            # Bullet-point findings grouped by layer

        def build_valuation_summary(self) -> list[Flowable]:
            # Key metrics table

        def build_dcf_projection(self) -> list[Flowable]:
            # Year-by-year DCF table

        def build_scenario_analysis(self) -> list[Flowable]:
            # Bull/Base/Bear scenario table

        def build_monte_carlo_chart(self) -> list[Flowable]:
            # Generate histogram with matplotlib → embed Image flowable

        def build_peer_comparison(self) -> list[Flowable]:
            # Peer multiples table

        def build_risk_factors(self) -> list[Flowable]:
            # Numbered risk list

        def build_disclaimer(self) -> list[Flowable]:
            # Legal text

        def generate(self, output_path: str) -> str:
            # Assemble all sections, build PDF, return file path
            doc = SimpleDocTemplate(output_path, pagesize=A4, ...)
            self.story = []
            self.story.extend(self.build_cover_page())
            self.story.extend(self.build_executive_summary())
            # ... all sections ...
            self.story.extend(self.build_disclaimer())
            doc.build(self.story)
            return output_path

  Styling:
    - Page size: A4 (210mm × 297mm)
    - Margins: 20mm all sides
    - Primary font: Helvetica (built into reportlab, no font files needed)
    - Headers: Helvetica-Bold, 14pt (section), 18pt (cover title)
    - Body text: Helvetica, 10pt
    - Table headers: Helvetica-Bold, 9pt, white on dark background
    - Table cells: Helvetica, 9pt, alternating row colors (#F5F5F5 / white)
    - Recommendation badge colors:
        BUY  → #1B5E20 (dark green) background, white text
        SELL → #B71C1C (dark red) background, white text
        HOLD → #E65100 (dark amber) background, white text
    - Footer: Page X of Y, generated timestamp, disclaimer snippet

10. RISKS & MITIGATIONS
================================================================================

  Risk 1: LLM Hallucination in Financial Numbers
    Mitigation: Every numeric output from an LLM agent is validated against
    known data sources (yfinance, SEC). The Valuation Engine uses only
    validated numbers. The LLM provides DIRECTIONAL adjustments to a base
    of hard financial data — it never generates financial numbers from scratch.

  Risk 2: DCF Terminal Value Dominance
    Mitigation: Cross-validate with relative valuation. Flag when TV >70% of
    total EV in the report. Use exit multiple method as sanity check.

  Risk 3: Garbage In, Garbage Out (Data Quality)
    Mitigation: Data quality scoring in confidence calculation. If financial
    data is sparse or inconsistent, confidence score drops and recommendation
    is caveated. SEC EDGAR serves as ground truth for US stocks.

  Risk 4: Model Drift Over Time
    Mitigation: Backtesting is core infrastructure (not a future enhancement).
    Every recommendation is tracked against actual 12-month price returns.
    Component-level attribution identifies which layer (growth, margin, WACC,
    terminal value) contributed most to forecast error. If recommendation
    accuracy degrades, recalibrate agent prompts and bucket boundaries.

  Risk 5: Rate Limits on Free APIs
    Mitigation: Aggressive caching. Financial data TTLs are long (daily to
    monthly). Cache hit rate should be >90% for repeat valuations.

  Risk 6: Unrealistic LLM-Generated Assumptions
    Mitigation: The Assumption Validation Layer (Section 5.8) caps all
    agent-generated assumptions at 2σ historical bounds. Any assumption
    exceeding historical ranges is flagged, capped, and reported in the
    PDF with transparency notes. Confidence score is penalized for RED flags.

11. SUCCESS METRICS

  Operational Metrics:
    - DCF model matches within ±10% of manual Excel DCF for same inputs
    - Agent-generated WACC components (CRP, beta) within ±20% of professional
      data sources (Damodaran, KPMG, Kroll)

  Backtesting-Driven Metrics (Core Infrastructure):
    - BUY recommendations outperform market by 5%+ over 12 months
    - Recommendations match professional analyst consensus direction >60% of time
      (disagreement is okay — the AI should find opportunities analysts miss)

  Component-Level Validation (Measurable per-component effectiveness):
    - Country Analysis: Does including CRP improve valuation accuracy vs.
      using a flat 5% ERP? (A/B test on historical basket)
    - Industry Analysis: Does industry-specific beta produce better forecasts
      than using a flat β=1.0?
    - Company Analysis: Does the management quality score correlate with
      subsequent earnings surprises?
    - Monte Carlo: Is the 10th-90th percentile range well-calibrated?
      (actual price should fall within the range ~80% of the time)
    - Relative Valuation: Does blending DCF with relative valuation (70/30)
      outperform pure DCF or pure relative valuation?
    - Discrete Buckets: Do the chosen bucket boundaries (e.g., CRP at
      0|250|500|750|1000|1500 bps) segment actual country returns correctly?

  Without these measurements, it is impossible to determine whether any
  component of the system actually creates value beyond a simple DCF with
  default assumptions. Backtesting is the only mechanism for validating
  that the top-down architecture, multi-agent design, and qualitative-to-
  quantitative conversion collectively produce better results than a
  naive spreadsheet model.

13. IMPLEMENTATION DEVIATIONS
================================================================================

The following deviations from the original plan were made during implementation.
Each is intentional and documented for transparency.

13.1 LangGraph Compatibility & Sequential Fallback
--------------------------------------------------------------------------------
  Plan:   LangGraph StateGraph compiles directly with ValuationState(dict).
  Actual: The custom dict subclass caused 'ticker' KeyError during invoke().
          Added a sequential fallback pipeline that runs each agent node
          function in order. The LangGraph graph is still built and attempted;
          if it fails, execution proceeds sequentially.

  Impact:  More robust — works regardless of LangGraph version. No loss of
           functionality since the pipeline was linear (no branching needed).

13.2 Raw sqlite3 Instead of sqlitedict
--------------------------------------------------------------------------------
  Plan:   Use sqlitedict library for application-level caching.
  Actual: Used Python's built-in sqlite3 module. sqlitedict was removed from
          requirements.txt and Claude.md dependency list.

  Why:    Eliminates a dependency. Gives finer control over WAL mode, busy
          timeout, table schema, stats tracking. The DataCache class is more
          featureful (hit/miss counters, explicit expire(), clear()).

13.3 Monte Carlo Iterations (5k vs 10k)
--------------------------------------------------------------------------------
  Plan:   10,000 iterations.
  Actual: Default 5,000 with comment: "5k for speed; use 10k for production".
          The MonteCarloSimulation class accepts iterations as a parameter.

  Why:    Faster CLI feedback during development. Users can pass --iterations
          for higher precision.

13.4 yfinance Field Name Aliasing
--------------------------------------------------------------------------------
  Plan:   Direct field name lookups in yfinance DataFrames (e.g.,
          "Operating Cash Flow").
  Actual: yfinance uses different field names across stocks and statement
          types. Added fuzzy-match alias dict in _get_latest() that maps our
          canonical names to multiple possible yfinance names.

  Example: "Operating Cash Flow" maps to ["Operating Cash Flow",
           "Cash Flow From Continuing Operating Activities",
           "Net Cash Provided By Operating Activities"].

13.5 No .env.example File
--------------------------------------------------------------------------------
  Plan:   Create .env.example as a template for required keys.
  Actual: Only .env exists (gitignored). Key documentation is in Section 3.1.

13.6 Agent Architecture Simplification
--------------------------------------------------------------------------------
  Plan:   Individual LangGraph agent classes in src/agents/ (country_agent.py,
          industry_agent.py, company_agent.py, etc.).
  Actual: Agent prompts in src/prompts/ as standalone modules. Agent logic
          (LLM calls, parsing, state updates) lives in orchestrator.py node
          functions. The src/agents/ directory has only __init__.py.

  Why:    Cleaner — one orchestrator to maintain vs 5 agent class files.
          The prompts are still modular and independently editable.

13.7 Shares Outstanding Units
--------------------------------------------------------------------------------
  Plan:   Shares outstanding in millions.
  Actual: yfinance returns raw share count (e.g., 14,687,356,000 for AAPL).
          The DCF model divides equity value by this raw count. Math is
          consistent across all calculations.
